# Универсальный шаблон ответа на билет (Node.js + Express + Sequelize + PostgreSQL + EJS)

Ниже — готовый «скелет ответа», который можно адаптировать почти под любой билет из ваших примеров (заявки, задачи, авто, объекты, кампании, сервисные заявки).

---

## 1) Короткий ответ для защиты (что сказать в начале)

> Я реализовал веб-приложение на **Node.js + Express** с базой **PostgreSQL** через **Sequelize ORM**.  
> Интерфейс сделан на **EJS + Bootstrap 5**.  
> Добавлены **аутентификация и авторизация** через `express-session` и `bcrypt` (хеширование паролей).  
> Система поддерживает: вход пользователя, просмотр списка сущностей, фильтрацию, CRUD для роли менеджера/админа, валидацию и обработку ошибок.

## 2) Как закрываются критерии из таблицы

### ПК 09.1–09.2 (Проектирование БД)

1. Выделена предметная область: основная сущность (например, `Request`, `Task`, `Car`, `Object`, `Campaign`) + справочники (`Status`, `Type`, `User` и т.д.).
2. Указаны атрибуты, ключи, типы:
   - PK: `id` (INTEGER, auto increment)
   - FK: `status_id`, `type_id`, `assignee_id`, `created_by`
   - даты: `DATE` / `TIMESTAMP`
   - деньги: `DECIMAL(10,2)`
   - тексты: `VARCHAR`, `TEXT`
3. Нормализация: вынесены повторяющиеся значения в отдельные таблицы (`statuses`, `types`, `roles`).
4. Связи:
   - `User (1) -> (M) MainEntity`
   - `Status (1) -> (M) MainEntity`
   - `Type (1) -> (M) MainEntity`

### ПК 09.3–09.4 (Интерфейс и логика)

1. Первая страница — форма входа.
2. После входа — список записей.
3. Реализован фильтр (по дате/статусу/типу/исполнителю и т.п.).
4. Для роли manager/admin: создание, редактирование, удаление.
5. Обратная связь: flash-сообщения/alert (`успешно`, `ошибка`).
6. Обработка ошибок: 404, 500, валидация формы.
7. Минималистичный UI на Bootstrap 5.

### ПК 09.5–09.6 (Безопасность)

1. `bcrypt` для паролей (в БД хранится только hash).
2. `express-session` для сессии пользователя.
3. Роли (`admin`, `manager`, `user`) и middleware-проверка доступа.
4. Запрет доступа к защищённым маршрутам без авторизации.

## 3) Универсальная структура БД (под любой билет)

### Таблица `users`
- `id` PK
- `full_name` VARCHAR(100)
- `login` VARCHAR(50) UNIQUE
- `password_hash` VARCHAR(255)
- `role_id` FK -> `roles.id`
- `created_at`, `updated_at`

### Таблица `roles`
- `id` PK
- `name` VARCHAR(30) UNIQUE (`admin`, `manager`, `user`)

### Таблица `statuses`
- `id` PK
- `name` VARCHAR(50) UNIQUE (например: `new`, `in_progress`, `done`, `rejected`)

### Таблица `main_entities` (универсальная)
- `id` PK
- `title` VARCHAR(150)
- `description` TEXT
- `start_date` DATE
- `end_date` DATE
- `price_or_budget` DECIMAL(10,2) NULL
- `type_id` FK -> `types.id`
- `status_id` FK -> `statuses.id`
- `assignee_id` FK -> `users.id` NULL
- `created_by` FK -> `users.id`
- `created_at`, `updated_at`

### Таблица `types`
- `id` PK
- `name` VARCHAR(50) UNIQUE

> Под конкретный билет просто переименовываете `main_entities` в `applications`, `tasks`, `cars`, `construction_objects`, `campaigns`, `service_requests`.

## 5) Универсальные маршруты (шаблон)

- `GET /login` — страница входа
- `POST /login` — проверка логина/пароля
- `POST /logout` — выход
- `GET /` или `/entities` — список
- `GET /entities/new` — форма создания (manager/admin)
- `POST /entities` — создать
- `GET /entities/:id/edit` — форма редактирования (manager/admin)
- `POST /entities/:id` — обновить
- `POST /entities/:id/delete` — удалить

Фильтрация в списке:
- `GET /entities?status_id=1&type_id=2&date_from=...&date_to=...&assignee_id=...`

## 6) Что говорить про безопасность

Кратко на защите:

> Пароли пользователей не хранятся в открытом виде — только `bcrypt`-хеши.  
> После входа создаётся серверная сессия (`express-session`), по которой проверяется доступ к закрытым страницам.  
> Реализованы роли: обычный пользователь видит список и фильтрацию, менеджер/админ может выполнять CRUD.

## 9) Быстрая адаптация под конкретный билет

- **Учет заявок на выставки**: сущность `Application`, поля: компания, тип выставки, дата участия, статус.
- **Проектное агентство**: `Task`, поля: заголовок, описание, срок, сложность, исполнитель, статус.
- **Автопрокат**: `Car`, поля: марка, модель, год, цена аренды, доступность.
- **Строительные объекты**: `ConstructionObject`, поля: название, даты, статус, подрядчик.
- **Рекламное агентство**: `Campaign`, поля: название, даты, аудитория, бюджет, канал, статус.
- **Сервисный центр**: `ServiceRequest`, поля: тип ремонта, описание, исполнитель, дата, статус.
